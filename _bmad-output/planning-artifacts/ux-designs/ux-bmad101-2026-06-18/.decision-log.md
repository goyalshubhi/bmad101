# Decision Log — Deck Generation UX (3-Screen Scope)

## 2026-06-18: Scope & constraints

- **Form factor:** Desktop-first only. Mobile deferred.
- **Protagonist:** FP&A/strategy analyst, messy data, tight timeline before leadership review. Speed + visible trust signals > hand-holding.
- **Screens in scope:** (1) Clarifying questions, (2) Narrative picker, (3) Audit/verification view.
- **Out of scope:** Visual identity (DESIGN.md), rendered deck output (PPTX/HTML), full design system.
- **Audit view mode:** Single view with two entry points — blocking gate after narrative selection (fix-it actions prominent) + read-only after render.
- **MVP bar:** Functional and demoable, not visually elaborate. Wireframe-level.

## 2026-06-18: Screen design decisions

- **Question interface:** One-at-a-time card (not all questions visible simultaneously). Rationale: analyst persona scans fast but needs the data observation context per question — stacking all five with their contexts would create wall-of-text. Answered stack below gives full visibility without clutter.
- **Suggestion chips → free-text seeding:** Chips populate the input rather than being selectable options. Preserves free-text flexibility while reducing typing for obvious answers.
- **Narrative cards → detail panel below:** Side-by-side cards for scan, single expandable panel for deep read. Not a two-column card+detail layout — screen width at 1280px doesn't support three cards + a detail pane simultaneously.
- **Audit view tabs (Figures / Checks / Assumptions):** Three concerns, one view. Tabs over sections because figures table can be long and tabs keep the mental model clean. Figures tab is default because it's the primary trust signal.
- **Gate status as sticky bottom bar:** Riya needs constant visibility of "what's still blocking me" while she works through fixes in any tab.
- **5 data gaps flagged:** assumption sign-off tracking, check dismissal tracking, fix-application endpoint, per-figure match status, assumption challenge round-trip.
